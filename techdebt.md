## 2026-06-10 | low | Recommendation copy lives in code constants, not strings.xml (B-048)

**Context:** The recommendations engine (`android/metrics/.../recommendation/RecommendationEngine.kt`) builds each tip's Spanish title/body inline in Kotlin, heavily parameterised with pesos/percentages/hour-blocks it computes. Only the section header (`recommendations_title`) and the premium gate hint (`gate_hint_recommendations`) are in `android/ui/.../res/values/strings.xml`. **Why deferred:** Extracting parameterised copy into `strings.xml` formatter args would split each rule across two files and complicate the pure-module boundary, for no user benefit at a single-locale (es-MX) launch. The copy is already driver-tone reviewed (tú, direct, no jargon).
**When to fix:** If/when Kompara localises beyond es-MX, or a designer/copy review wants the strings centralised — move the rule bodies to `strings.xml` with positional formatter args and have the `:ui` layer format them (the engine would then return a copy key + args instead of a finished string).

## 2026-06-10 | High | Platform-withholding rate estimates need counsel verification — LAUNCH GATE for the fiscal feature (B-052)

**Context:** The monthly fiscal summary + PDF export (B-052) estimates what Uber/DiDi/inDrive withhold under the Mexican "régimen de plataformas tecnológicas". All rates are data-driven constants in ONE file — `android/metrics/.../fiscal/PlatformWithholdingRates.kt` — with legal citations inline: **ISR 2.1% of gross** (LISR art. 113-A fr. These were verified June 2026 via WebSearch against the SAT minisitio + practitioner guidance (siemprealdia.co, FacturoPorTi, grupofiscalcastroycia, consultoriofiscal.unam.mx) but are **research-grade, not confirmed by counsel/an accountant**. Modelling assumptions baked in: (1) every platform/driver is the *registered-RFC, terrestrial-passenger-transport* case — a driver without an RFC faces full-16% IVA withholding and possibly different ISR, and delivery vs. rides differs; (2) the withholding base is the **gross we observe** (captured offer-fare estimates / imported gross), not the platform's CFDI base, so centavos won't tie out; (3) DiDi reports no commission split, so its fiscal *net* is approximated as = gross and flagged (the *withholdings* are on gross and unaffected). **Why deferred:** No authoritative SAT/counsel feed in this environment to confirm against; the values are correct enough to *build and test* the feature, and they're isolated in one constants file so a correction is a one-line edit (no logic change). The honest disclaimers keep the estimates from being mistaken for a fiscal document pre-review.

**When to fix:** **BEFORE the fiscal feature is exposed publicly** (this is a launch gate). Have counsel / an accountant confirm: (a) the 2.1% ISR + 8% IVA rates for transport-with-RFC for the current year; (b) whether to model the no-RFC (full-16% IVA) and delivery cases per-driver instead of assuming registered-transport; (c) whether the "pago definitivo" framing in the CSF explainer is accurate. ## 2026-06-10 | low | Referral device-fraud heuristic is coarse (one redemption per device id) (B-056)

**Context:** Redemption abuse control (`backend/src/routes/referrals.ts`) rests on: one redemption per account (unique `redeemer_driver_id`), one redemption per device (unique `redeemer_device_id`), a self-referral check, and a < 30-day account-age requirement. The device id is the app-generated anonymous install UUID sent by the client — it is **not** an attested hardware identifier, so a determined abuser can reset it (reinstall / clear data) to free up a device for another redemption, and the account-age + per-account caps still bound the payoff. There is no velocity limiting per referrer, no IP/ASN signal, and no manual review queue.
**Why deferred:** The current caps make farming low-yield (each fake account needs a fresh install AND is < 30 days old AND nets one 14-day grant) and avoid taking a dependency on Play Integrity / device attestation in v1. Good enough to launch the loop.
**When to fix:** If abuse appears in the attribution data — add referrer-side velocity limits, consider Play Integrity attestation for the redeemer device, and add a review/clawback path for suspicious clusters.
