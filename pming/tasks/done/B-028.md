# Declarative parser framework & fixture harness

Parsers as data, not code: a spec engine that matches offer cards and extracts fields from node-tree snapshots, with a fixture-based regression harness. This is the single highest-leverage architectural piece (host UI churn must be fixable without a Play release).

## What to do

1. `:parsers` — parser spec model (JSON/kotlinx-serialization): card detector (anchor patterns), field extractors (anchored regex/selector chains over flattened tree), unit normalizers (es-MX numbers, currency, km/min), versioned per `(targetPackage, versionCode range)`
2. Spec engine: evaluate spec against snapshot → typed `OfferCard` (fare, pickupDistance, pickupTime, tripDistance, tripTime, surge, variant) or no-match
3. Extraction rules: never absolute child index or raw coordinates; text anchors + role + bounds as signals
4. Fixture harness: record/replay sanitized snapshots as JSON fixtures; JUnit parameterized tests run every spec against its fixture corpus
5. PII scrubbing for fixtures (strip names, plates, addresses beyond zone)

## Acceptance criteria
- A spec change requires zero Kotlin changes to ship new extraction logic
- Fixture tests fail loudly when a spec regresses
- Engine handles multiple card variants (surge, multi-stop, reservation) per platform
