# Verdict chip: net $/hr row + two-tier semáforo thresholds

The collapsed chip showed net $/km plus the net trip total ("ganancia neta") — but the trip total
duplicates the fare the host app already shows. Replace it with net $/hr (the second number the
verdict is judged on), and replace the single-floor verdict with explicit yellow/red thresholds.

## What was done
1. `VerdictChipState`/`VerdictChipUi`: net $/hr ("$186/h", whole pesos) directly under the $/km
   hero; "ganancia neta" moved into the expanded detail rows.
2. `PlatformThreshold` gained red floors (`redPerKmMxn`, `redPerHourMxn`), defaulting to 75% of the
   green floors so existing installs and the city seed table migrate free.
3. `NetProfitEngine.decideLevel`: per-metric two-tier classification (≥ green → GREEN, < red → RED,
   between → YELLOW); overall verdict GREEN only if all testable metrics green, RED only if all
   red, YELLOW for any mix. Missing-fare → RED and the partial-data GREEN→YELLOW cap unchanged.
4. Long-press quick sheet: two $/km sliders ("Verde desde" / "Rojo por debajo de") with red ≤ green
   enforced in `ThresholdSheet`; persisted via `SettingsRepository` (new red-floor DataStore keys,
   backward-compatible decode).
5. Simulator "bad" demo fixtures worsened so they still read RED inside the new yellow band.

## Acceptance criteria
- ✅ Chip shows $/km hero + $/hr beneath; net total only in expanded detail
- ✅ Verdict: green at/above green floor, red below red floor, yellow between
- ✅ Existing persisted single-floor settings decode with derived red floors
- ✅ All unit tests green (`metrics`, `data`, `overlay`, `ui`, `parsers`)
