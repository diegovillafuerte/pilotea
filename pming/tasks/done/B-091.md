# Play-compliance hardening — kill the #1 launch risk

The top-ranked launch risk (per the 2026-06-18 triple-review) is **Google Play killing the app
on policy grounds**: commercial screen-reading via AccessibilityService + overlay is the exact
heuristic signature Play Protect uses for banking trojans. The mitigation posture is locked:
read-only forever, everything on-device, nothing stored or transmitted from the screen, clean
declaration. Most of it already exists (disclosure hard gate B-036, `isAccessibilityTool="false"`,
in-memory frames, debug-gated fixtures). This task closes the three remaining gaps:

1. **Read-only is convention, not structure.** Nothing stops a future change from adding
   `performAction`/`dispatchGesture`. Add a build-time guard (`verifyReadOnlyCapture`) wired into
   `check`/CI that fails the build on any automation API in production source, and asserts the
   accessibility-service config invariants (`isAccessibilityTool="false"`, no action capabilities).

2. **The capture lane still rides MediaProjection everywhere.** The locked 2026-06-18 architecture
   (StopClub/Brazil-validated) is a11y-detect + silent `AccessibilityService.takeScreenshot()`
   (API 30+) + on-device OCR + `TYPE_ACCESSIBILITY_OVERLAY` chip — one declared lane, no
   per-session projection consent, no cast indicator, survives screen lock, and drops the
   FGS-mediaProjection declaration burden on 30+ devices. Implement it as the primary lane on
   API 30+, with automatic fallback to the existing MediaProjection flow (<30, or when an OEM
   breaks `takeScreenshot`).

3. **No privacy policy / declaration texts for the new lane.** Draft the privacy policy and update
   the Play declaration + Data Safety drafts to the takeScreenshot story (counsel review stays
   B-038).

## Acceptance criteria

- `./gradlew verifyReadOnlyCapture` fails if `performAction(`, `dispatchGesture(`,
  `performGlobalAction(` or `GLOBAL_ACTION_` appear in any module's `src/main`; passes today.
  Runs in CI.
- Guard also fails if the a11y config loses `isAccessibilityTool="false"` or gains
  `canPerformGestures`/`flagRequestFilterKeyEvents`/`flagRequestTouchExploration`.
- On API 30+, enabling the accessibility service is sufficient for the reader: offers OCR'd from
  silent screenshots while a target app is foreground; frames in-memory only; recycled after OCR.
- MediaProjection flow unchanged on API <30 and used as automatic fallback when the screenshot
  lane errors repeatedly on-device.
- No double-capture: the projection consent path no-ops while the screenshot lane is active.
- Unit tests for the new gating/failure logic; all existing tests green; lint clean.
- Privacy policy draft + updated declaration texts in docs/; design doc reflects the locked lane.

## Out of scope

- On-device validation (founder, S25) — added to techdebt as the flip-gate.
- Counsel review of copy (B-038).
- Removing the MediaProjection path entirely (it stays as <30 fallback).

## Resolution — DONE (2026-07-02)

Shipped on `feat/b091-play-compliance-lane`. All acceptance criteria met; Gradle build, full
`testDebugUnitTest`, and `verifyReadOnlyCapture` green; `:ocr`/`:capture`/`:data`/`:app` lint clean
(pre-existing `:ui` lint errors untouched).

- **Read-only guard:** `android/build.gradle.kts` `verifyReadOnlyCapture` — fails on any
  `performAction(`/`dispatchGesture(`/`performGlobalAction(`/`GLOBAL_ACTION_` in any production source
  set (main/release/flavors, not test), and on loss of `isAccessibilityTool="false"` / addition of a
  gesture-or-key capability. Wired into every module's `check` and into Android CI. Verified it passes
  clean, catches an injected token, and greens on removal.
- **Silent lane:** `SilentScreenshotLane` (`:ocr`) — API 30+ `AccessibilityService.takeScreenshot` →
  in-memory ML Kit OCR → shared `OfferFramePipeline`, gated on the LIVE `rootInActiveWindow` package so
  it only ever captures a target host app. Bound via `ScreenReaderLane` (interface in `:capture`,
  `@Provides` in `:app`), started in `onServiceConnected`. `OcrCaptureService` (MediaProjection) stays
  the <30 lane + automatic fallback; both feed the same extracted `OfferFramePipeline` (no drift).
  `OcrConsentActivity` no-ops while the silent lane is active (no double capture).
- **Docs:** `docs/play-compliance.md` (canonical posture), `docs/privacy-policy.md` (DRAFT, B-038),
  design-doc §2.1, and the launch-runbook Play/Data-Safety drafts updated to the takeScreenshot story.
- **Adversarial review:** two codex rounds → 8 issues found and fixed (unrelated-app capture window,
  bitmap recycle-on-cancellation, running-state ownership, teardown reset race across loop generations,
  resume-drop leak, chip-rect timing, hardware-bitmap copy-throw leak, guard source-set scope).
- **On-device validation owed:** TD-040 (takeScreenshot cadence / hw-buffer conversion / OEM fallback).
