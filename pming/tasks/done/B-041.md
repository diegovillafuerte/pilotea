# Greenfield backend scaffold

The thin new backend (full greenfield per pivot decision): auth, consented aggregate ingestion, benchmarks delivery, parser-config hosting, telemetry ingestion, subscription state. Small, boring, managed.

## What to do

1. Choose and scaffold the stack (implementation decision — bias to boring: managed Postgres + thin typed API service; containerized; deployable on Render/Fly/Railway). Document the choice in `docs/android-technical-design.md` §4
2. Schema v1: drivers, sessions, weekly_aggregates (per driver × platform × week), population_stats (port web schema), parser_configs, telemetry_counters
3. API skeleton: health, auth endpoints (stub), aggregates upsert, benchmarks fetch, config fetch (signed)
4. CI + deploy pipeline + .env.example; secrets never committed
5. Port the `get_percentile` SQL function and population_stats seed data from the web repo (reference reuse encouraged — deployment is new, logic is proven)

## Acceptance criteria
- Deployed health endpoint; CI green
- Schema migrations reproducible from scratch
- Percentile function returns same results as web MVP for identical inputs (port test)
