# Onboarding & permission walkthrough

The make-or-break funnel: value pitch → prominent disclosure → accessibility grant → OEM keep-alive → first verdict. Competitors prove drivers will grant the permission if the value is shown first.

## What to do

1. Onboarding flow (Compose): 3-screen value pitch (verdicts before accepting; "+30%" framing; "tus datos nunca salen de tu teléfono")
2. **Prominent disclosure screen** per Play policy: standalone, plain Spanish, states the app reads the Uber/DiDi screen content on-device to compute trip metrics; explicit "Aceptar" required before any Settings deep-link
3. Accessibility grant walkthrough: deep-link + animated step guide; detect grant success and celebrate
4. OEM survival kit: detect Xiaomi/Oppo/Vivo/Samsung; tailored autostart/battery whitelisting steps (dontkillmyapp patterns); lock-in-recents tip
5. Service watchdog: notification + in-app banner when the service gets disabled/killed, one-tap re-enable path
6. ToS risk disclosure copy (from B-038) embedded in disclosure flow

## Acceptance criteria
- Disclosure meets Play prominent-disclosure standard (screenshot evidence for the declaration form)
- Grant-success detection works; funnel events logged (anonymized)
- Per-OEM steps shown only on matching devices

## Dependencies
- B-026 (design system), B-027 (service to enable), B-038 (legal copy)

## Completion note (2026-06-10)

Implemented the full funnel in `:ui` as its own nav graph (`OnboardingNavGraph`), routed by a new
`KomparaRoot` that observes `Settings.onboardingCompleted` (persisted via `SettingsRepository`):

- **Value pitch** (3-page `HorizontalPager`): verdicts-before-accepting, "+30%" framed honestly as
  "promedio reportado por apps similares", "tus datos nunca salen de tu teléfono".
- **Prominent disclosure** (standalone, scrollable, Play-structured): what is read / why / on-device /
  ToS-risk + explicit "Aceptar y continuar" vs "No acepto" (→ limited-info screen). Decline never
  deep-links to Settings. `DisclosureStateMachine` enforces the gate.
- **Accessibility grant**: stepwise instructions + deep-link to `ACTION_ACCESSIBILITY_SETTINGS`;
  observes `ServiceStatusProvider` (new `:data` abstraction bound to `:capture`'s
  `ServiceStateRepository`, keeping `:ui` off `:capture`) and auto-advances with a celebratory state
  when the service flips connected.
- **OEM survival kit**: `OemDetector` maps `Build.MANUFACTURER` (xiaomi/redmi/poco, oppo/realme/oneplus,
  vivo/iqoo, samsung, huawei/honor → tailored steps; generic otherwise); battery deep-link is the
  info-only `ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS` (never requests the exemption) + lock-in-recents tip.
- **Done**: "Probar el lector" routes to the simulator if its route exists in the graph (runtime
  check) else the Lector tab; requests `POST_NOTIFICATIONS` contextually on 33+.
- **Watchdog**: Hilt `ServiceWatchdog` (started from `KomparaApplication`) observes the service after
  onboarding completes; on a healthy→dropped edge it posts a notification on the "estado del lector"
  channel (guarded by the runtime permission) and exposes an in-app banner state the Inicio screen shows.
- **Funnel**: `OnboardingFunnel` interface → DataStore counters + Logcat (no network).
- Tests: routing, OEM mapping table, watchdog transitions, funnel recording, disclosure accept/decline.
