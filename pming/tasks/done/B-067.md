# Required signup: phone → WhatsApp OTP → perfil

B-042 built the backend (drivers table with UUID, WhatsApp OTP, sessions, device merge) and the
`AuthRepository`, but the app had no account UI and was anonymous-first. Product direction changed
(StopClub/GigU model): drivers must create an account.

## What was done
1. New `mx.kompara.ui.auth` package: `SignupFlowScreen` (phone → 6-digit WhatsApp code → profile
   name/ciudad), `SignupViewModel` driving the steps against `AuthRepository`, `MxPhone` E.164
   normalization (accepts bare 10 digits, `+52`, legacy `+521`), resend cooldown, 401-aware error
   mapping.
2. Onboarding funnel: SIGNUP step inserted after the pitch (new `OnboardingStep.SIGNUP`/
   `SIGNUP_DONE` funnel counters); profile city mirrors into `SettingsRepository.setCity` so
   benchmarks use it without asking twice.
3. Root gate: `RootRouter` now also requires an authenticated session — completed-onboarding
   installs without a session land on a standalone signup (`RootRoute.AUTH`).
4. Copy: native es-MX auth strings; updated the now-false "funciona sin cuenta" claims in the
   pitch and prominent disclosure.

## Acceptance criteria
- ✅ New install cannot reach the main shell without verifying a phone via WhatsApp OTP
- ✅ Anonymous device id merges into the account on first verify (backend B-042 path)
- ✅ Re-entry mid-flow resumes at the profile step instead of re-asking the phone
- ✅ All unit tests green
