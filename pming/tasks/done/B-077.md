# Chip latency: faster first verdict, no mid-offer dropout

On-device feedback (2026-06-12): the chip can take seconds to appear on a DiDi offer, and sometimes
appears immediately but then hides while the card is still on screen. Root causes: 500 ms OCR
cadence bounds first-parse latency, and the offer-gone heuristic counts every unparseable frame —
the animating map under the card garbles OCR often enough that 6 consecutive misses (~3 s) happen
mid-offer.

## What to do
1. OCR cadence 500 ms → 300 ms (faster first verdict AND faster true hide)
2. Only count a frame toward "offer gone" when it lacks the card signature (fare + leg pattern);
   garbled-but-card-like frames keep the chip up
3. 15 s no-successful-parse zombie backstop so a stuck signature can never pin a stale verdict
4. Extract the decision into a pure, unit-tested tracker

## Acceptance criteria
- Chip stays up through OCR garble while the card is visible (signature frames never hide it)
- Card dismissal still hides the chip in ~2 s
- Tracker covered by JVM tests (garble run, dismissal, zombie timeout)

## Result
Shipped 2026-06-12 (PR #6, commits 09eccf2 + 4c81bec + 767e68c). 300 ms OCR cadence;
CardPresenceTracker holds the verdict through garbled-but-card-like frames (15 s zombie backstop);
KomparaUiGuard stops self-parsing (found live: Inicio stats parsed as a DiDi fare). Real root cause
of the mid-offer dropout found with live offers: the node path emitted spurious NoCard for DiDi
(no text in its Flutter tree) racing the OCR verdict on the shared bus — node path now forwards
nothing for OCR-owned platforms, and OCR re-asserts Parsed every frame so any stray hide self-heals
in ~300 ms. Verified on real offers in Cuernavaca: chip appeared ~1 s and held through full offers.
