# Core architecture: modules, DI, Room, DataStore

Lay down the module structure and core infrastructure per `docs/android-technical-design.md` §3.

## What to do

1. Gradle modules: `:capture`, `:parsers`, `:overlay`, `:metrics`, `:data`, `:sync`, `:ui`, `:billing` (empty but wired)
2. Hilt DI across modules
3. `:data` — Room database: `offers`, `trips`, `shifts`, `cost_profile` entities + DAOs; DataStore for settings (thresholds, platform toggles)
4. Kotlin coroutines/Flow conventions; dispatcher injection for testability
5. Unit test scaffolding per module (JUnit5 or JUnit4+Robolectric where needed)

## Acceptance criteria
- Modules compile with enforced boundaries (`:ui` cannot depend on `:capture` internals)
- Room schema exported; migration test harness in place
- Sample end-to-end DI graph proves wiring (app launches with injected no-op services)
