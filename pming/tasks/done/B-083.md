# Ledger verdict ignores configured per-platform thresholds

**Found on-device 2026-06-15.** The verdict persisted with each offer is computed against the
**default** thresholds, not the driver's **configured** per-platform thresholds — so the ledger logs a
different traffic-light color than the overlay showed the driver. In this session, **6 of 8 offers
diverged**.

## Root cause
```kotlin
// TripLifecycleTracker.evaluateVerdict — TripLifecycleTracker.kt:111
return engine.evaluate(offer, costProfile, PlatformThreshold.DEFAULT).verdict.level.name
```
It hardcodes `PlatformThreshold.DEFAULT` (8.0/km green, 6.0 red; 90/h green, 67.5 red), while the
overlay/metrics path uses the configured per-platform thresholds from the settings datastore. On the
test device those were Uber & DiDi `10.0/7.5 $/km`, `154/115.5` and `120/100 $/h` — far stricter than
default — so the ledger systematically **over-rates** rides (biases history green).

|---|---|---|---|
| Uber 57.34 (id4) | 9.25 | YELLOW | **GREEN** |
| DiDi 49.77 (id6) | 7.06 | RED | **YELLOW** |
| DiDi 79.52 (id7) | 7.32 | RED | **YELLOW** |
| DiDi 61.52 (id8) | 8.73 | YELLOW | **GREEN** |
| DiDi 65.44 (id9) | 9.28 | YELLOW | **GREEN** |
| DiDi 65.02 (id10) | 6.63 | RED | **YELLOW** |
| Uber 243.71 (id5) | 5.37 | RED | RED (agree — both extremes) |

All 6 mid-range offers diverge: the ledger `verdict` column matches the DEFAULT computation exactly
(confirming the model), and is one rung greener than the configured-threshold color the driver saw.
(Uber id3 also diverges but is conflated with the OCR garble — see B-081.) Pull the DB WITH its `-wal`
file (WAL mode) or these rows look stale.

## Impact
The paid history/benchmarks layer (the monetized product) will tell the driver their rides were better
than the reader warned — the moment they compare the live overlay verdict against their logged history,
trust breaks. Any per-platform threshold tuning the driver did is silently ignored in storage.

## What to do
- Resolve the driver's per-platform `PlatformThreshold` (the same values the overlay/metrics engine
  uses, from the settings repository) and pass it into `engine.evaluate` instead of
  `PlatformThreshold.DEFAULT`. Also pass the configured `PreferredMetric` if the overlay does.
- Test: an offer that is GREEN under default but YELLOW/RED under a stricter configured threshold
  persists the configured color.

## Result — code complete, merged to `main` (PR #10, commit 7dcf29b)
Fixed. `TripLifecycleTracker.evaluateVerdict` now reads `settingsRepository.settings.first()` and grades
with `effectiveThreshold` + `preferredMetric` (the exact inputs `OverlayController` grades the live chip
with) instead of `PlatformThreshold.DEFAULT`. Injected `SettingsRepository` into the tracker; added
`testImplementation(libs.androidx.datastore.preferences)` to `:capture` so the test drives a real
temp-file DataStore. Two regression tests added: a 9.0 $/km card persists **YELLOW** under a stricter
configured floor (10.0/7.5) and **GREEN** under the default — proving the divergence is closed.
`:capture:testDebugUnitTest` green. Resolves TD-025.

**Codex adversarial review (2026-06-18, commit 7dcf29b): confirmed** — `evaluateVerdict` (`TripLifecycleTracker.kt:102-122`)
passes `settings.effectiveThreshold` + `settings.preferredMetric` into `engine.evaluate`; **no
`PlatformThreshold.DEFAULT` path remains** in the method. Verdict: SAFE-TO-VERIFY-ON-DEVICE.
