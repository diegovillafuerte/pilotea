# Driver-feedback UX polish: verdict copy, metric labels, simulator parity, Lector revamp

Juan's on-device feedback after B-079 (2026-06-12): the verdict wording is redundant ("verde"
already means it pays — drop "conviene / no conviene"), IPK/IPH are internal vocabulary drivers
don't know, the simulator copy reads stale against the preferred-metric changes, and the Lector
screen doesn't clearly explain what each connection is for. User first.

## What to do
1. Verdict copy: remove the redundant "conviene / no conviene" phrasing wherever the colour
   already says it (chip label line, simulator headlines, anywhere else a grep finds) — keep the
   copy clean and minimal; colour does the talking
2. Metric labels: drop "(IPK)" / "(IPH)" from all driver-facing strings (settings toggle, hints);
   the enum stays internal
3. Simulator: coherence pass vs B-079 — stale km-centric copy (intro, why-narratives, metric rows)
   must follow or acknowledge the driver's chosen metric
4. Lector screen revamp: one clear card per connection — (a) accessibility reader (Uber) and
   (b) screen-capture reader (DiDi) — each with: what it is for in one plain-Spanish line, live
   status, and its single action; keep the prominent-disclosure dialog; keep simulator + semáforo
   shortcuts visible but secondary
5. Full-app cohesion audit vs the latest changes (B-066/070/074/075/076/079): find and fix stale
   copy or visual inconsistencies; list anything deliberately deferred

## Acceptance criteria
- No driver-facing "conviene/no conviene" redundancy and no IPK/IPH abbreviations anywhere in the UI
- Simulator copy is coherent with the preferred-metric model under both IPK and IPH
- Lector screen shows two distinct connection sections with status + one action each, in plain
  Spanish, no jargon
- All Android unit tests pass; new/changed copy covered where tests exist
