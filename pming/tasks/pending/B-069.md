# Account management in Ajustes + session-expiry re-auth

Signup is required (B-067) but has no mirror surface: the driver can't see their account, edit
their profile, log out, or delete the account — and an expired 30-day session has no re-auth path.

## What to do
1. Ajustes "Tu cuenta" section: phone (read-only), name/city editing (PATCH /v1/me), cerrar sesión.
2. Delete-account flow (Play data-safety requires it before submission) — needs the matching
   backend endpoint.
3. 401 handling: when a bearer call fails with an expired session, clear local auth so the root
   gate flips to the signup flow instead of silently failing.
4. Standalone AUTH flow currently skips the profile step (root flips to MAIN on verify) — decide
   whether to keep that or hold the gate until the flow completes.

## Acceptance criteria
- Driver can view/edit profile, log out, and delete the account from Ajustes
- Expired session lands on signup, not on broken screens

## Result

Implemented across backend + Android.

**Item 1 — "Tu cuenta" in Ajustes:** new `AccountScreen` + `AccountViewModel` (route `account`, launcher
button at the top of Ajustes). Shows the read-only WhatsApp phone, editable name + benchmark city
(PATCH /v1/me, mirroring the city into settings exactly like the signup profile step), a "Guardar
cambios" CTA, "Cerrar sesión", and "Eliminar mi cuenta" with a confirm dialog.

**Item 2 — Delete-account flow:** new `DELETE /v1/me` backend endpoint (hard-delete; every child FK is
ON DELETE CASCADE except devices which is SET NULL so the anonymous reader survives). `ApiClient.deleteMe()`
+ `AuthRepository.deleteAccount()` (deletes server-side then clears local auth). Backend tests cover
delete + session revocation, device set-null survival, and the 401-without-bearer case.

**Item 3 — 401 re-auth:** added a `SessionInvalidator` callback to `ApiClient`; strictly bearer-required
calls (`getMe`/`getMeFull`/`updateMe`/`deleteMe`/`pushAggregate`/`importWeek`/`syncSubscription`/referral
reads+redeem) route through `ensureOkAuthed`, which clears the persisted token + cached driver on a 401.
That flips `SessionState` to Anonymous, so the root gate routes to AUTH (signup) instead of every authed
call silently failing. Anonymous-allowed endpoints (benchmarks/configs/telemetry) keep `ensureOk` so a
signed-out 401 never wipes a still-valid session. The DI provider writes the same DataStore keys directly,
keeping the graph acyclic.

**Item 4 — DECISION: hold the gate until the standalone flow completes.** Rationale: name/ciudad feed the
benchmarks, so skipping the profile step on the standalone path loses data we want. Implemented as a one-shot
latch in `KomparaRoot` (`authFlowActive`) plus a pure `RootRouter.effectiveRoute(route, authFlowActive)`
helper (unit-tested): once we enter AUTH this session we keep rendering the signup flow until its
`onComplete` fires (profile saved or explicitly skipped via "Completar después"), even though the session —
and thus the route — flips to MAIN the instant the OTP verifies. A returning, already-authenticated user
(latch off) still goes straight to MAIN.

### Pending on-device verification (not possible in this harness)
- "Tu cuenta": load profile, edit name/city + save, cerrar sesión lands on signup, delete + confirm lands on signup.
- Standalone AUTH path (completed-onboarding install, no session): phone → code → PROFILE step now shows before MAIN.
- 401 re-auth: with an expired/revoked session, the next authed call drops to the signup gate (no broken screens).
