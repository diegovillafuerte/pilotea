# Repair corrupt ledger data from node-path clock & OCR decimal bugs

On-device testing (2026-06-15, Samsung S25, CDMX) surfaced two ledger-corrupting bugs whose
**root causes are now fixed in the working tree** (pending review/ship). This task cleans up the
**already-persisted bad rows** they left behind — the code fixes stop new corruption but do not
repair history.

## The bad data (observed on-device)
- **74 phantom trips** in `trips`, all `grossMxn=0 / distanceKm=0 / durationMin=0`, `estimated=1`,
  no linked `offerId`, with `startedAt`/`endedAt` stored as `SystemClock.uptimeMillis` (~243M ms →
  render as Jan 1970). 68 UBER / 6 DIDI. Created by an earlier build where the node path classified
  Uber's (no-longer-node-readable) offer screens as trip/idle. The source is already closed (Uber +
  DiDi are now in `OCR_OWNED_PACKAGES`), but the rows remain and will skew daily/weekly aggregates.
- **Offer `id=3`** persisted `pickupKm=4.0 / distanceKm=4.8` from a first-frame OCR decimal-loss
  (`"0.4 km"` → `"04 km"` → `4.0`); true values pickup 0.4 km / trip 0.8 km (total 1.2 km). The
  overlay self-corrected on the next frame but `OfferSeen` fires once, so the ledger froze the garble.

## Root-cause fixes already made (verify they landed first)
- `capture/.../KomparaAccessibilityService.kt:170` — node-path captures now stamp
  `System.currentTimeMillis()` instead of `event.eventTime` (uptime). Stops new 1970 timestamps and
  the clock-domain mix in the merged `TripLifecycleTracker` stream.
- `ocr/.../UberOcrParser.kt` `cleanNumber` — recovers the leading-zero decimal-loss (`"04"` → `"0.4"`).
  Test: `UberOcrParserTest."recovers a pickup distance whose decimal point OCR dropped"`.

## What to do
1. Write a Room migration (or one-off repair on next launch, idempotent) that **deletes** the
   zero-value phantom trips: `estimated=1 AND grossMxn=0 AND distanceKm=0 AND durationMin=0 AND
   offerId IS NULL`. Recompute affected daily/weekly aggregates afterward (trip count, hours).
2. Decide on the uptime-dated rows generally: any `trips`/`offers`/`shifts` row whose timestamp is
   implausibly small (e.g. `< 10^12`, i.e. before 2001) is a node-path uptime artifact — drop or
   quarantine; do not try to back-convert (the boot epoch is unknown retroactively).
3. Offer `id=3`: it is a declined/expired single offer; simplest is to leave it (parser fix prevents
   recurrence) or null out the distance fields. Document the choice.

## Acceptance criteria
- After migration: no `trips` rows with a pre-2001 `startedAt`; phantom zero-value trips gone.
- Daily/weekly aggregates recomputed and consistent (no orphaned trip counts/hours).
- Migration is idempotent and backward-compatible (passes `/migration-check`).

## Related findings from the same session
- **B-083 (high)** — ledger verdict uses `PlatformThreshold.DEFAULT`, not the configured per-platform
  thresholds, so persisted verdicts diverge from what the overlay showed (6 of 8 offers this session).
- **DiDi parser has the same latent decimal-loss class** — `DidiOcrParser` cleans numbers inline (not
  via a shared `cleanNumber`), so a DiDi km leg `"0.4"` → `"04"` would read 4.0. Protected here only
  incidentally (short pickups render in meters). Port Uber's `cleanNumber` + leading-zero guard.
- **Ledger freezes the first parsed frame** — `OcrLifecycleClassifier` emits `OfferSeen` once per
  session and deliberately does not re-key on distance; a garbled first frame sticks even when later
  frames are clean (this is what corrupted offer id=3). The parser fix narrows the leading-zero case
  but doesn't fully close it. Consider updating the persisted offer if a later same-session frame
  yields a materially different, more-complete parse.
