# Verdict overlay UI

The floating verdict drivers see over the host app: traffic light + key numbers, rendered as a `TYPE_ACCESSIBILITY_OVERLAY` window from the service (no SYSTEM_ALERT_WINDOW permission, immune to `setHideOverlayWindows`).

## What to do

1. `:overlay` — Compose-in-window overlay attached from the AccessibilityService
2. Verdict chip: traffic light (verde/amarillo/rojo) + net $/km + $/min + estimated net profit for the trip; expandable to detail row
3. Positioning: draggable, snap-to-edge, position persisted; auto-hide when no card on screen; never obstruct the host app's Accept button area (configurable safe zones)
4. Sunlight-readable, glanceable typography (driver is in motion — 1-second cognition target)
5. Overlay lifecycle bound to card visibility events from `:capture`
6. Quick-settings inline: tap-and-hold to adjust threshold without leaving the host app

## Acceptance criteria
- Card appears → verdict visible < 150 ms end-to-end (with parser)
- Never intercepts touches outside its own bounds
- Survives orientation/IME changes; no leaks on service restart

## Dependencies
- B-027 (service hosts the overlay window)
