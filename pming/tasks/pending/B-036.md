# Onboarding & permission walkthrough

The make-or-break funnel: value pitch → prominent disclosure → accessibility grant → OEM keep-alive → first verdict. Competitors prove drivers will grant the permission if the value is shown first.

## What to do

1. Onboarding flow (Compose): 3-screen value pitch (verdicts before accepting; "+30%" framing; "tus datos nunca salen de tu teléfono")
2. **Prominent disclosure screen** per Play policy: standalone, plain Spanish, states the app reads the Uber/DiDi screen content on-device to compute trip metrics; explicit "Aceptar" required before any Settings deep-link
3. Accessibility grant walkthrough: deep-link + animated step guide; detect grant success and celebrate
4. OEM survival kit: detect Xiaomi/Oppo/Vivo/Samsung; tailored autostart/battery whitelisting steps (dontkillmyapp patterns); lock-in-recents tip
5. Service watchdog: notification + in-app banner when the service gets disabled/killed, one-tap re-enable path
6. ToS risk disclosure copy (from B-038) embedded in disclosure flow

## Acceptance criteria
- Disclosure meets Play prominent-disclosure standard (screenshot evidence for the declaration form)
- Grant-success detection works; funnel events logged (anonymized)
- Per-OEM steps shown only on matching devices

- B-026 (design system), B-027 (service to enable), B-038 (legal copy)
