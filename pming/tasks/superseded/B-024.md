# Trip broadcast card parser with efficiency metrics

Parse real-time trip broadcast screenshots (the card a driver sees before accepting) and return actionable efficiency metrics: earnings per km and earnings per hour including pickup deadhead.

## Context

Drivers get a broadcast card when a trip is offered. The card shows:
- **Fare** (e.g. $105.84)
- **Surge/bonus** (e.g. $3.23)
- **Rider rating and trip count** (e.g. 5.00, 4 viajes)
- **Pickup ETA and distance** (e.g. 4min, 693m)
- **Trip duration and distance** (e.g. 25min, 14.9km)
- **Pickup address**
- **Dropoff address**
- **Verified payment badge** (tarjeta bancaria verificada)

This is fundamentally different from the weekly summary parsers (B-007, B-012) — it's per-trip, real-time decision support.

## What to do

1. Create `src/lib/parsers/trip-broadcast.ts` with a new parser that uses Claude Vision to extract fields from a trip broadcast screenshot.
2. Define a `TripBroadcastResult` type with:
   - `fare`: number (total fare shown)
   - `surge_bonus`: number | null
   - `rider_rating`: number | null
   - `rider_trips`: number | null
   - `pickup_eta_min`: number (pickup time in minutes)
   - `pickup_distance_km`: number (pickup distance in km — convert from meters if needed)
   - `trip_duration_min`: number (trip time in minutes)
   - `trip_distance_km`: number (trip distance in km)
   - `pickup_address`: string | null
   - `dropoff_address`: string | null
   - `payment_verified`: boolean
   - `platform`: 'didi' | 'uber' | 'indrive' (detect from UI style)
3. Compute derived efficiency metrics:
   - `earnings_per_km`: `fare / (pickup_distance_km + trip_distance_km)` — total distance including deadhead
   - `earnings_per_hour`: `fare / ((pickup_eta_min + trip_duration_min) / 60)` — total time including pickup
   - `dead_km_ratio`: `pickup_distance_km / (pickup_distance_km + trip_distance_km)` — what % of driving is unpaid
4. Add Zod validation schema for the Vision extraction response.
5. Handle unit conversions: pickup distance may be in meters (e.g. "693m") while trip distance is in km (e.g. "14.9km"). Normalize everything to km.
6. Write unit tests with mocked Vision responses covering:
   - DiDi broadcast card (like the sample image)
   - Edge cases: missing surge, very short pickup, distance in different units

## Acceptance criteria

- Parser extracts all visible fields from a DiDi trip broadcast screenshot
- `earnings_per_km` correctly includes pickup deadhead distance
- `earnings_per_hour` correctly includes pickup time
- `dead_km_ratio` tells driver what fraction of driving is unpaid
- Unit conversion handles m→km correctly (693m → 0.693km)
- Zod schema validates Vision response before metric computation
- Tests pass with mocked Vision responses
- For the sample image: earnings_per_km ≈ $6.79/km, earnings_per_hour ≈ $219/hr

## Dependencies

- Existing Claude Vision client (`src/lib/parsers/utils.ts`)
