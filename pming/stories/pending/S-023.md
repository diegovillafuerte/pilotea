# OCR capture for SurfaceView platforms

On-device verification (2026-06-11, see `docs/android-technical-design.md` §7) proved DiDi and inDrive render their entire UI on a SurfaceView and expose **zero text** to accessibility — the node-tree reader (S-012) reads Uber only. To read DiDi/inDrive we capture the screen via MediaProjection and run on-device OCR (ML Kit), turning recognized text + bounds into the same `OfferCard` the rest of the pipeline already consumes. DiDi offer cards are not FLAG_SECURE (driver screenshots work), so capture + OCR is viable — this is how Ruta Rentable reads DiDi in MX.

## Tasks
- B-059: MediaProjection capture service & consent
- B-060: On-device OCR engine (ML Kit)
- B-061: Region-based OCR parser specs
- B-062: Wire OCR into the verdict pipeline
- B-063: DiDi MX OCR spec (real captures)
- B-064: inDrive MX OCR spec (real captures)
- B-065: Play declaration & data-safety for MediaProjection

## Acceptance criteria
- Live DiDi + inDrive offer cards produce correct overlay verdicts via OCR
- On-device only: captured frames never leave the phone; nothing persisted beyond debug fixtures
- Per-session consent + foreground-service capture indicator handled per Android 14/15 policy
- Unified `OfferEvent` output so overlay / trip log / telemetry work unchanged across capture modes
